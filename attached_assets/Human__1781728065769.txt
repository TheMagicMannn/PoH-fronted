1. PURPOSE
The Human Authenticity Intelligence Engine determines whether a session, visitor, device, or interaction originates from a real human, a suspicious human, an automated system, an emulator, a bot, a farmed account, or a synthetic/abusive actor.
The engine is designed to power:
	•	Human verification
	•	Bot detection
	•	Automation detection
	•	Fraud scoring inputs
	•	Trust scoring inputs
	•	Revenue protection decisions
	•	Session risk classification
	•	Challenge/allow/block decisioning
The engine must work across:
	•	Browser SDK
	•	Server SDK
	•	Mobile SDK
	•	API-based scoring
	•	Batch replay / retrospective analysis
 
⸻
 
2. ENGINE OBJECTIVES
Primary Objective
Produce a calibrated Human Authenticity Score from 0 to 100 with a confidence value and supporting reasons.
Secondary Objectives
	•	Detect non-human or low-human-likelihood behavior
	•	Provide interpretable reason codes
	•	Support real-time decisioning with low latency
	•	Minimize false positives on legitimate users
	•	Adapt to changing automation patterns over time
	•	Operate with privacy-aware collection controls
 
⸻
 
3. SCOPE
In Scope
	•	Human score generation
	•	Behavior feature extraction
	•	Browser integrity validation
	•	Headless and automation detection
	•	Interaction rhythm analysis
	•	Session-level classification
	•	Device and environment signal analysis
	•	Human authenticity explanations

	•	Feedback ingestion for model improvement
Out of Scope

⸻
 
4. DEFINITIONS
Human Authenticity
The likelihood that the entity generating the session is a real person operating a non-automated browser or app environment.
Session
A contiguous sequence of user interactions tied to one visitor/device context.
Visitor
A logical user identity, potentially anonymous, stitched across sessions.
Device
A browser, mobile device, emulator, or other client environment.
Automation
Scripted, programmatic, or bot-like behavior including headless browsers, automation frameworks, scripted clicks, or synthetic interaction patterns.
Confidence
A value between 0.0 and 1.0 indicating how stable and supported the score is.
Reason Code
A machine-readable explanation for why the score is high or low.

5. OUTPUT CONTRACT
5.1 Human Authenticity Score
Range: 0-100
Interpretation:
	•	90-100: Strong human likelihood
	•	70-89: Likely human with some anomalies
	•	40-69: Suspicious / mixed signals
	•	0-39: Strong automation or non-human likelihood
5.2 Human Confidence
Range: 0.0-1.0
Interpretation:
	•	<0.50: Low confidence
	•	0.50-0.75: Moderate confidence
	•	0.75: High confidence
5.3 Human Classification
One of:
	•	human
	•	suspicious_human
	•	automation
	•	bot
	•	emulator
	•	uncertain
5.4 Recommended Decision
One of:
	•	allow
	•	challenge
	•	step_up
	•	block
5.5 Reason Codes
A bounded enum of machine-readable explanations.

7. SIGNAL COLLECTION REQUIREMENTS
7.1 Collection Principles
	•	Collect only what is necessary for legitimate fraud prevention and trust scoring.
	•	Respect consent and jurisdictional restrictions.
	•	Avoid persistent identifiers beyond the platform’s stated purpose.
	•	Support graceful degradation when signals are unavailable.
7.2 Required Client Permissions
The SDK must function even if some browser features are blocked. The engine must rely on signal fusion rather than any single fingerprint field.
7.3 Collection Timing
Signals should be collected at:
	•	session start
	•	first interaction
	•	form interaction
	•	significant user events
	•	pre-decision moments
	•	heartbeat intervals

10. SCORING ARCHITECTURE
10.1 Architecture Overview
The engine uses a multi-stage scoring pipeline:
	1.	Signal validation
	2.	Feature normalization
	3.	Feature group scoring
	4.	Model inference
	5.	Calibration
	6.	Rule overlay
	7.	Confidence estimation
	8.	Decision mapping
 
⸻
 
10.2 Feature Group Weights
Suggested default weights:
	•	Browser Integrity: 25%
	•	Human Behavior: 25%
	•	Device Consistency: 15%
	•	Network Reputation: 20%
	•	Historical Authenticity: 15%
Weights should be configurable per customer tier and use case.
 
⸻
 
10.3 Final Human Authenticity Score Formula
final_score =
  0.25 * browser_integrity_score +
  0.25 * human_behavior_score +
  0.15 * device_consistency_score +
  0.20 * network_reputation_score +
  0.15 * historical_authenticity_score
Then apply calibration and policy adjustments.
 
⸻
 
10.4 Confidence Estimation
Confidence should be reduced when:
	•	too many signals are missing
	•	the session is too short
	•	the device is unseen and uncorrelated
	•	the network is noisy or masked
	•	the model has low support for the observed pattern
Confidence should be increased when:
	•	multiple independent signals agree
	•	historical reputation exists
	•	interaction volume is sufficient
	•	behavior and device data are stable
Suggested confidence formula:
confidence = signal_completeness * model_agreement * historical_support * calibration_quality
Values must be clamped to 0.0–1.0.
 
⸻
 
11. DECISION LOGIC
11.1 Decision Thresholds
Default thresholds:
	•	allow: >= 85
	•	step_up: 65-84
	•	challenge: 40-64
	•	block: < 40
These thresholds must be customer-configurable.
11.2 Decision Rules
A score alone should not decide every case.
Examples:
	•	High score + high confidence => allow
	•	Moderate score + high transaction value => step_up or challenge
	•	Low score + proxy + automation => block
	•	Low score + missing signals + low confidence => challenge rather than block
11.3 Decision Hierarchy
Hard blocks may override soft score outputs when severe rules trigger:
	•	confirmed automation signatures
	•	high-confidence datacenter proxy abuse
	•	repeated synthetic behavior
	•	known malicious device cluster

17. MODELING APPROACH
17.1 Recommended Model Stack
Use a layered ensemble:
	•	rules layer for hard signals
	•	gradient-boosted model for structured features
	•	sequence model for session behavior
	•	calibration layer for score normalization
17.2 Candidate Model Types
	•	XGBoost / LightGBM for tabular signals
	•	Logistic regression for calibration
	•	Sequence classifier for session rhythms
	•	Outlier detection for anomaly flags
	•	Similarity model for device/session clustering
17.3 Feature Normalization
All features must be transformed into comparable numeric ranges:
	•	binary -> 0/1
	•	probabilities -> 0-1
	•	counts -> log-scaled
	•	time intervals -> bucketed or normalized
	•	categorical -> encoded

18. RULES ENGINE INTEGRATION
The engine must expose feature and reason outputs to the policy engine.
Example rules

	•	challenge if human_score < 70 and confidence < 0.80
	•	allow if human_score > 90 and trust_score > 85
The rules engine should be able to override or amend model outputs.

29. SUMMARY
The Human Authenticity Intelligence Engine is a multi-signal, real-time classification system that converts device, behavior, network, and historical features into a human authenticity score, confidence value, classification, and decision. It is designed to be explainable, configurable, low-latency, and extensible enough to power the PoH Intelligence platform at scale.
